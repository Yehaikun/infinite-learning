## 1. 构建上下文深度理解

### 1.1 什么是构建上下文

构建上下文是Dockerfile所在目录及其子目录，docker build命令会将这个目录打包发送给Docker daemon。理解构建上下文对于优化构建速度和避免安全问题至关重要。

**一句话解释**：构建上下文就是"打包发送到Docker daemon的文件夹"，这个文件夹包含了构建镜像所需的所有文件。

### 1.2 构建上下文的工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                   构建上下文工作流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 执行 docker build -t myapp .                           │
│                    ↑                                       │
│                    └─ . 是构建上下文                        │
│                                                             │
│  2. Docker客户端                                            │
│     └── 打包整个上下文目录（除了.dockerignore）              │
│                                                             │
│  3. 发送到Docker daemon                                    │
│     └── 上传打包的文件                                       │
│                                                             │
│  4. Dockerfile中的COPY/ADD指令                              │
│     └── 从打包的文件中复制到镜像                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 1.3 构建上下文示例

假设项目结构：

```
myproject/
├── Dockerfile
├── src/
│   ├── main.py
│   └── utils.py
├── tests/
│   └── test.py
├── .git/
├── node_modules/      # 可能有几百MB
├── .dockerignore
└── README.md
```

执行构建：

```bash
cd myproject
docker build -t myapp .
```

**执行结果**：

```
Sending build context to Docker daemon  523.4MB   ← 注意！整个目录都打包了
Step 1/1 : FROM ubuntu:22.04
 ---> 8a3cdc4d1ad3
Successfully built 8a3cdc4d1ad3
Successfully tagged myapp:latest
```

**问题**：即使Dockerfile只有一行`FROM ubuntu`，node_modules（500MB）也会被发送到Docker daemon，导致构建非常慢！

---

### 1.4 .dockerignore的高级用法

**.dockerignore文件的作用**：排除不需要打包到构建上下文的文件

**创建.dockerignore**：

```dockerfile
# .dockerignore

# Git相关
.git
.gitignore

# 文档
*.md
docs/

# Python缓存
__pycache__
*.pyc
*.pyo
*.pyd

# Node.js
node_modules/
npm-debug.log

# 测试和覆盖率
tests/
coverage/
.pytest_cache/

# IDE
.vscode/
.idea/
*.swp

# 环境文件
.env
.env.local
secrets.json

# 日志
*.log

# 操作系统文件
.DS_Store
Thumbs.db
```

**执行效果对比**：

```

# ✅ 有.dockerignore
Sending build context to Docker daemon  2.3MB    ← 减少了99%！
```

## 2. 多阶段构建原理

### 2.1 什么是多阶段构建

多阶段构建允许在一个Dockerfile中使用多个FROM指令，每个FROM创建一个构建阶段，最终镜像只包含最后一个阶段的内容。

**一句话解释**：多阶段构建就是"分步建造"，先用一个阶段来编译/构建，然后用另一个干净的小镜像来运行。

---

### 2.2 单阶段vs多阶段对比

**单阶段构建（镜像大）**：

```dockerfile
FROM node:18

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build
```

**执行结果**：

```
docker build -t myapp:big .
Sending build context to Docker daemon  45.2MB
Step 1/5 : FROM node:18
 ---> 8a3cdc4d1ad3
Step 2/5 : WORKDIR /app
 ---> Running in 9a5b4c8e1234
...
Successfully built abc123def456
Successfully tagged myapp:big

docker images myapp:big
REPOSITORY   TAG       SIZE
myapp        big       1.12GB    ← 镜像非常大！
```

---

**多阶段构建（镜像小）**：

```dockerfile
# 阶段1：构建
FROM node:18 AS builder

WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# 阶段2：运行（只有几十MB）
FROM node:18-alpine

WORKDIR /app

COPY --from=builder /app/dist ./dist

EXPOSE 3000

CMD ["node", "dist/index.js"]
```

**执行结果**：

```
docker build -t myapp:small .
Sending build context to Docker daemon  45.2MB
Step 1/9 : FROM node:18 AS builder
 ---> 8a3cdc4d1ad3
Step 2/9 : WORKDIR /app
 ---> Running in 9a5b4c8e1234
...
Step 5/9 : FROM node:18-alpine
 ---> 4d2c1b0e8f7a     ← 开始第二阶段
...
Successfully built abc123def456
Successfully tagged myapp:small

docker images myapp:small
REPOSITORY   TAG       SIZE
myapp        small     178MB    ← 镜像小很多！
```

---

### 2.3 多阶段构建的优势

| 方面 | 单阶段 | 多阶段 |
|------|--------|--------|
| 镜像大小 | 1-2GB | 100-300MB |
| 构建时间 | 较长 | 较短（利用缓存） |
| 安全性 | 低（包含构建工具） | 高（只包含运行时） |
| 攻击面 | 大 | 小 |

**执行对比**：

```
# 单阶段
docker images | grep myapp
myapp  big      1.12GB

# 多阶段  
docker images | grep myapp
myapp  small    178MB
镜像体积减少了 84%！
```

---

### 2.4 多阶段构建语法

```dockerfile
# 定义阶段
FROM <image> AS <stage-name>

# 引用阶段
COPY --from=<stage-name> <src> <dest>
```

**执行示例**：

```dockerfile
# 给阶段起名字
FROM golang:1.21 AS builder

# 后面引用这个阶段
COPY --from=builder /app/main .
```

**执行结果**：

```
Step 7/10 : COPY --from=builder /app/main .
 ---> Using cache   ← 使用了builder阶段的缓存！
 ---> abc123def456
```

## 3. 构建缓存优化策略

### 3.1 缓存机制详解

Docker使用层缓存来加速构建，每条指令的执行结果会被缓存：

```
┌─────────────────────────────────────────────────────────────┐
│                     缓存命中流程                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1/4: FROM ubuntu:22.04                              │
│  ├─ 本地有缓存 → 使用缓存 abc123                           │
│  └─ 无缓存 → 拉取镜像                                       │
│                                                             │
│  Step 2/4: RUN apt-get update                            │
│  ├─ 本地有缓存 → 使用缓存 def456                           │
│  └─ 无缓存 → 执行命令并缓存                                 │
│                                                             │
│  Step 3/4: RUN apt-get install -y nginx                  │
│  ├─ 本地有缓存 → 使用缓存 ghi789                           │
│  └─ 无缓存 → 执行命令并缓存                                 │
│                                                             │
│  Step 4/4: COPY . /app                                    │
│  ├─ 文件变化 → 重新执行（不使用缓存）                       │
│  └─ 后续层也重新执行                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**第一次构建**：

```
docker build -t myapp .
Sending build context to Docker daemon  1.2MB
Step 1/4 : FROM ubuntu:22.04
 ---> 8a3cdc4d1ad3
Step 2/4 : RUN apt-get update
 ---> Running in abc123def456
Get:1 http://archive.ubuntu.com/ubuntu jammy InRelease [25.6 MB]
...
 ---> 1a2b3c4d5e6f
Step 3/4 : RUN apt-get install -y nginx
 ---> Running in 2b3c4d5e6f7g
...
 ---> 3c4d5e6f7g8h
Step 4/4 : COPY . /app
 ---> 4d5e6f7g8h9i
Successfully built 4d5e6f7g8h9i
Successfully tagged myapp:latest
```

**第二次构建（无变化）**：

```
docker build -t myapp .
Sending build context to Docker daemon  1.2MB
Step 1/4 : FROM ubuntu:22.04
 ---> 8a3cdc4d1ad3  Cached   ← 使用缓存！
Step 2/4 : RUN apt-get update
 ---> 1a2b3c4d5e6f  Cached   ← 使用缓存！
Step 3/4 : RUN apt-get install -y nginx
 ---> 3c4d5e6f7g8h  Cached   ← 使用缓存！
Step 4/4 : COPY . /app
 ---> 4d5e6f7g8h9i  Cached   ← 使用缓存！
Successfully built 4d5e6f7g8h9i
Successfully tagged myapp:latest
```

**第三次构建（修改了代码）**：

```
docker build -t myapp .
Sending build context to Docker daemon  1.2MB
Step 1/4 : FROM ubuntu:22.04
 ---> 8a3cdc4d1ad3  Cached
Step 2/4 : RUN apt-get update
 ---> 1a2b3c4d5e6f  Cached
Step 3/4 : RUN apt-get install -y nginx
 ---> 3c4d5e6f7g8h  Cached
Step 4/4 : COPY . /app
 ---> 5e6f7g8h9i0j  Changed   ← 文件变化，重新执行！
 ---> 6f7g8h9i0j1k
Successfully built 6f7g8h9i0j1k
Successfully tagged myapp:latest
```

---

### 3.2 利用缓存的最佳实践

**原则：变化少的放上面，变化多的放下**

**❌ 反模式**：

```dockerfile
FROM node:18
COPY . .                    # 代码每次都变，放上面
RUN npm ci                  # 依赖不常变，但放下面
RUN npm run build
```

**执行效果**：

```
# 第一次构建
Step 3/5 : COPY . .
 ---> 3c4d5e6f7g8h
Step 4/5 : RUN npm ci
 ---> 4d5e6f7g8h9i
Step 5/5 : RUN npm run build
 ---> 5e6f7g8h9i0j

# 修改代码后重新构建（npm ci又要重新执行！）
Step 3/5 : COPY . .
 ---> 6f7g8h9i0j1k  Changed
Step 4/5 : RUN npm ci        ← 重新执行！（慢）
 ---> 7g8h9i0j1k2l
Step 5/5 : RUN npm run build
 ---> 8h9i0j1k2l3m
```

---

**✅ 最佳实践**：

```dockerfile
FROM node:18
COPY package*.json ./      # 依赖文件先复制（不常变）
RUN npm ci                  # 安装依赖
COPY . .                    # 代码后复制（常变）
RUN npm run build
```

**执行效果**：

```
# 第一次构建
Step 3/5 : COPY package*.json ./
 ---> 3c4d5e6f7g8h
Step 4/5 : RUN npm ci
 ---> 4d5e6f7g8h9i
Step 5/5 : COPY . .
 ---> 5e6f7g8h9i0j
Step 6/5 : RUN npm run build
 ---> 6f7g8h9i0j1k

# 修改代码后重新构建（npm ci使用缓存！）
Step 3/5 : COPY package*.json ./
 ---> 3c4d5e6f7g8h  Cached
Step 4/5 : RUN npm ci        ← 使用缓存！（快）
 ---> 4d5e6f7g8h9i  Cached
Step 5/5 : COPY . .
 ---> 7g8h9i0j1k2l  Changed
Step 6/5 : RUN npm run build
 ---> 8h9i0j1k2l3m
```

**不使用缓存**：

```bash
docker build --no-cache -t myapp .
```


### 4.1 Java Spring Boot多阶段构建

**完整Dockerfile**：

```dockerfile
# ============ 构建阶段 ============
FROM eclipse-temurin:21-jdk AS builder

WORKDIR /app

# 先复制pom.xml（利用Maven缓存）
COPY pom.xml .
RUN mvn dependency:go-offline -B

# 再复制源码
COPY src ./src

# 构建
RUN mvn package -DskipTests -B

# ============ 运行阶段 ============
FROM eclipse-temurin:21-jre-alpine

WORKDIR /app

# 只复制jar（很小）
COPY --from=builder /app/target/*.jar app.jar

# 非root用户
RUN addgroup -g 1000 appgroup && \
    adduser -u 1000 -G appgroup -D appuser && \
    chown -R appuser:appgroup /app
    
USER appuser

# JVM参数
ENV JAVA_OPTS="-Xms256m -Xmx512m"

EXPOSE 8080

HEALTHCHECK --interval=60s --timeout=10s --start-period=30s --retries=3 \
    CMD wget --no-verbose --tries=1 --spider http://localhost:8080/actuator/health || exit 1
```

## 7. 本章总结

### 构建上下文要点

1. **构建上下文是打包发送到Docker daemon的文件夹**
2. **使用.dockerignore排除不需要的文件**（可减少99%的打包体积）
3. **考虑使用子目录作为构建上下文**

### 多阶段构建要点

1. **使用多阶段构建大幅减小镜像体积**（可达80%+）
2. **先构建后运行，分离构建环境和运行环境**
3. **使用--from引用前一阶段**

### 缓存优化要点

1. **变化少的指令放上面**（依赖文件先COPY）
2. **先复制依赖文件，再复制代码**（npm ci使用缓存）
3. **利用--cache-from手动管理缓存**

## 8. 课后问题

1. **构建上下文是什么？为什么重要？**
   - 构建上下文是docker build发送的整个目录
   - 决定哪些文件被打包，影响构建速度和安全性
   - 实践：500MB的node_modules会被打包，导致构建慢

2. **多阶段构建的优势是什么？**
   - 镜像更小（1GB → 200MB，减少80%+）
   - 更安全（不包含构建工具）
   - 构建更快（可利用缓存）

3. **如何优化Docker构建缓存？**
   - 变化少的指令放上面
   - 先复制依赖文件，再复制代码
   - 使用--cache-from指定缓存源

### 11.2 缓存问题排查

**场景：依赖没有更新**

```bash

Step 4/6 : RUN npm ci
 ---> Using cache   ← 使用了旧的缓存！
```

**问题**：修改了package.json，但构建使用了缓存

**原因**：npm ci使用了缓存，但实际需要重新安装

**解决方案**：强制不使用缓存

```bash

# 方案3：使用buildkit
DOCKER_BUILDKIT=1 docker build --no-cache-filter=runner -t myapp:1.0 .
```

## 14. 本章总结

### 构建上下文总结

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 构建太慢 | 打包文件太多 | 使用.dockerignore |
| 镜像太大 | 单阶段构建 | 使用多阶段构建 |
| 缓存失效 | 指令顺序不对 | 先复制依赖文件 |
| 构建失败 | 文件找不到 | 检查.dockerignore |

### 多阶段构建总结

1. **阶段1：构建** - 安装工具，编译代码
2. **阶段2：运行** - 复制必要文件，使用轻量镜像
3. **--from=builder** - 从构建阶段复制文件

### 缓存优化总结

```
最佳顺序：
1. 基础镜像（不变）
2. 系统依赖（不常变）
3. 项目依赖（不常变）
4. 源码（常变）
5. 构建命令（常变）
```

1. **构建上下文打包了500MB但实际只需要2MB，怎么解决？**
   - 使用.dockerignore排除不需要的文件

2. **每次修改代码后npm ci都要重新执行，很慢，怎么优化？**
   - 先COPY package*.json，再RUN npm ci，再COPY源码

3. **多阶段构建最终镜像包含JDK，很大，怎么减小？**
   - 使用多阶段构建，运行阶段只用JRE
